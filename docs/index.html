高考知识库是 Agent 的事实底座

这一页对应 PRD 的“高考知识库搭建”和技术架构中的数据层、检索层。它回答的是： Agent 做决策时，事实从哪里来，查询如何发生，为什么它不是单纯靠模型记忆在回答问题。

2900+ 高校核心信息

与模型的关系

大模型负责理解需求、组织推理和生成表达；知识库与工具层负责提供确定性事实、检索结果和计算结果。 也就是说，模型是“大脑”，但不是“数据库”。

知识库层

沉淀高考场景里的核心数据资产，支持 Agent 在不同决策任务里调取同一套权威底座。

  • 考试院、教育部、省招办、高校官网
  • 院校基础信息、专业设置、历年分数线
  • 就业质量报告、城市产业与政策信息

工具与检索层

把 Agent 的自然语言意图转成可执行动作，完成查数、过滤、计算和排序。

  • 知识库调用与结构化检索
  • 规则过滤与约束匹配
  • 冲稳保计算与候选排序

模型依赖层

模型并不替代知识库或工具，而是负责调度它们，把结果组织成完整决策链条。

  • 需求理解与任务拆解
  • 调用工具获取事实
  • 基于结果生成解释与建议
用户问题 例如：山东物化生 612 分，适合哪些计算机方向志愿？
Agent 发起调用 把需求拆成分数、选科、偏好、策略等查询条件
知识库与检索返回 拉回院校、专业、分数线、限制条件等结构化结果
工具计算 做过滤、冲稳保打标、候选排序与适配分析
回给 Agent 把确定性结果交回 Agent 进入后续决策与表达
这一页刻意不展开“多源校验、OCR+人工校验、分级采信、风险提示”这些内容，因为它们属于 `Tab 4｜可信校验`； 也不展开“报告预览、PDF 下载、重点标注”等交付能力，因为它们属于 `Tab 5｜报告交付`。
目标：把系统从“能回答”提升到“可用于高风险决策” 拆分：数据治理机制 / 运行时校验 / 风险暴露与回流

数据治理机制

这是系统上线前的长期底座治理，确保进入 Agent 的事实本身尽量可靠、及时、可审计。

  • 多源交叉比对
  • OCR + 人工抽检校验
  • 分级采信策略
  • 政策与院校信息定时更新

运行时校验

这是执行过程中对具体方案做的即时检查，避免把“看起来合理”的错误答案交付给用户。

  • 分数线与关键数字核验
  • 选科 / 身体条件 / 调剂限制校验
  • 冲稳保梯度合理性检查
  • 志愿排序逻辑检查

风险暴露与回流

系统不仅要发现问题，还要把不确定性显式暴露出来，并把问题带回后续修正流程。

  • 高风险项标注
  • “仅供参考”与绝对化表述约束
  • 替代方案补充
  • 问题清单回流反思阶段
Checkpoint 1 关键数据先比对来源一致性，再进入候选方案计算。
Checkpoint 2 候选志愿生成后，立刻检查选科、身体条件与调剂限制。
Checkpoint 3 方案成型后检查冲稳保梯度，避免整体过冲或整体过保。
Checkpoint 4 发现问题不直接输出，而是进入风险提示与补方案流程。

平衡

Agent 实时生成过程

等待启动。点击左侧按钮后，系统会模拟规划、检索、校验和报告组装。

01
任务规划 判断当前策略应优先保专业、保学校，还是拉开冲稳保梯度。
02
知识库检索 拉取虚构样例中的院校、专业、分数线和限制条件。
03
冲稳保计算 根据风险偏好对候选学校分层，并生成解释理由。
04
可信校验 检查专业限制、地域偏好和整体梯度是否存在明显风险。
05
报告组装 输出策略摘要、志愿样例、风险提醒和下一步操作建议。
